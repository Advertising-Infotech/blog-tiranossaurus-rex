# Session 000048 — Fix infinite scroll pagination: news were repeating instead of loading older posts

## Date
2026-06-28

## Task
"Ver mais notícias" button kept repeating the same news instead of showing older posts from the database of 1163 scraped news.

## Root cause analysis
Three interrelated bugs:

### Bug 1 — `data-total="0"` (index.php:3)
`$GLOBALS['wp_total_posts']` was set inside `have_posts()`, but the `data-total` attribute was rendered BEFORE `have_posts()` was called. The JS received `total=0`, which made `totalItems` default to `items.length` (the number of initially rendered cards), causing the JS to think all posts were already loaded.

### Bug 2 — 27 initial posts (wp-load.php:158)
`have_posts()` sliced 27 posts from the JSON (`array_slice($all_posts, 0, 27)`). The JS expected 12 per page (`itemsPerLoad=12`). After showing 27 posts (12 + 12 + 3), the API was called at page 2 which returned IDs 13-24 — already shown from the initial 27 batch. Results: visible duplication on each "Ver mais" click.

### Bug 3 — Dead end after hiddenItems exhausted
With `data-total="0"`, after all 27 initial posts were shown:
```
hiddenItems.length === 0 && items.length >= totalItems
⇒ 0 === 0 && 27 >= 27 ⇒ true
```
This hid the "Ver mais" button permanently — the API was NEVER called.

## Changes made

### Fix 1 — wp-load.php (line 158)
Changed `array_slice($all_posts, 0, 27)` → `array_slice($all_posts, 0, 12)`. Initial HTML now renders exactly one page of posts (matching JS `itemsPerLoad=12`), eliminating overlap with API page 2 (IDs 13+).

### Fix 2 — index.php (line 1-6)
Moved `have_posts()` call before `data-total` attribute:
```php
get_header();
have_posts();
$total_posts = $GLOBALS['wp_total_posts'] ?? 0;
```
Now `data-total="1163"` (the actual total), enabling proper pagination.

### Fix 3 — No JS changes needed
With 12 initial posts + ~2 ad cards = 14 total items, only 2 are hidden. After showing them, API page 2 starts at ID 13 (confirmed: offset 12, IDs 13-24). No overlap.
